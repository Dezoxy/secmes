# Review 03 — Auth, identity & device trust

> **Slice 3** of the security review campaign (`docs/planning/security-review-campaign-plan.md`).
> **Date:** 2026-06-19 · **Reviewed against `main`** (post-#252).
> **Method:** ultracode workflow — `crypto-reviewer` + `security-boundary-auditor` subagents (Opus, max effort): 1 recon + 6 adversarial finders (one per claim, each trying to *break* it) + a skeptic refutation pass on every finding.
> **Result:** 4 claims **PROVEN**, 1 **PARTIAL** (`session-token-integrity` — security-critical vectors all held; two literal clauses overstated). 12 confirmed/downgraded findings, all **P3** — **except F1 on `fingerprint-binding`, which the skeptic *upgraded* in impact** (the device-linking OOB code is a brute-forceable ~30-bit backstop). No P1/P2 *forgery/privesc/cross-tenant* break surfaced, but the enrollment-code finding is the one behaviour-touching item that needs its own fix PR. The headline question — **can a malicious server silently add a device?** — is answered: **no**, the silent-add path is cryptographically closed.

## Claims in scope (proves invariants 1, 2, 3, 4, 6 on the auth/identity surface)

| # | Claim | Verdict |
|---|---|---|

## Evidence (highlights — full per-claim evidence in the workflow transcript)

- **session-token-integrity (PARTIAL).** Forgery blocked: `verify()` pins `algorithms:['EdDSA']` (`auth.service.ts:31,50`) and the verify key is public-only — `session-key.config.ts:40-42` reimports `{kty,crv,x}` with no private scalar `d`, so even an in-memory key read cannot HMAC-sign; `iss='argus'`/`aud='argus-api'` enforced (`auth.service.ts:48-49`). Prod fails closed if `SESSION_SIGNING_KEY_FILE` unset (`session-key.config.ts:48-52`) — key is a Key Vault credential file, never env (invariant #5). Refresh = `randomBytes(32)` 256-bit CSPRNG (`session-token.service.ts:42,75,140`), stored SHA-256-hex only (`0031_auth_sessions.sql:20`); access token never persisted. Replay of the **refresh** chain is closed: rotated-reuse → full-family revocation (`session-token.service.ts:119-133`), rotation-race → optimistic lock + family revoke (`:163-190`). Tenant binding is unforgeable — derived from `user_tenant_index` keyed on the signature-verified `sub`, never a JWT claim (`auth.service.ts:66-78`); the table is `SELECT`+`INSERT` only (no `UPDATE`, `0018_tenant_onboarding.sql:14,25`) and every upsert is `onConflictDoNothing` → write-once. **Two clauses overstated:** (a) a *revoked* session's already-minted **access** token stays cryptographically valid on non-admin routes for its full 10-min TTL — only `AdminGuard` re-checks `revoked_at` (ST-1); (b) the token is **not** device-bound — claims are `{sub,sid,uid}` and `auth_sessions` has no `device_id` (ST-2).
- **passkey-webauthn.** Challenge is server-issued 32-byte CSPRNG (`webauthn.service.ts:126,324`), stored keyed by a 122-bit random ceremony id (`0033_webauthn.sql:42`), single-use via `DELETE…RETURNING` with row-level locking so exactly one concurrent verifier wins and a replay → `UnauthorizedException` (`:203-214`/`:366-377`). The deleted bytes are the `expectedChallenge`; `@simplewebauthn/server@13.3.1` enforces equality vs `clientDataJSON.challenge`, plus the `type` (`webauthn.create`/`webauthn.get`) check blocks cross-ceremony reuse. `expectedRPID`/`expectedOrigin` + `requireUserVerification:true` on every verify (`:254-255`/`:425-426`); prod boot fails on `rpID==='localhost'` (`:81-85`). Identity is resolved **only** from the stored credential row matched by `credentialId` (`:389-409`); client `userHandle` is cross-checked but never authoritative (`:414-420`). Clone/counter: delete-on-use primary, BigInt regression gate (`:449-454`) + optimistic-lock embedding the stored counter in the `UPDATE … WHERE` (`:459-468`, race-loser audited `passkey.counter_regression`). Logging is pseudonymous `argusId=` only; non-`HttpException` library errors remap to 401 (`:306,:501`) so malformed bodies yield a controlled 401, not a crash.
- **breakglass-fenced.** **Both** guards apply: class-level `@UseGuards(CfAccessGuard)` (`breakglass.controller.ts:75`) fronts login **and** rotate; rotate adds `@UseGuards(AdminGuard)` (`:130`). The controller spec pins this via `Reflector.getAllAndMerge` (`breakglass.controller.spec.ts:47,57`), so dropping either guard fails a unit test. Forged CfAccess JWT rejected — `jwtVerify` pins issuer+audience+`algorithms:['RS256']` against remote JWKS, any failure → generic 401 (`cf-access.guard.ts:69-77`); header read server-side, cloudflared strips client copies. `AdminGuard` fails closed on no tenant, re-checks `sid` revocation (closing the 10-min window after rotate), requires `role='admin'`+`status='active'` (`admin.guard.ts:20,27-38,40-56`); rotate also rejects any session without an Argus `userId` (`:153`) so an OIDC admin token can't rotate. **Invariant 6 holds** — the only admin routes return device **key prefix** (`left(...,12)`, `admin.service.ts:59`), device delete, and audit metadata bounded by strict Zod (no content field); grep for `ciphertext|content|privateKey|sessionKey|attachment` across `admin/`+`breakglass/` → zero content paths. Audit un-skippable: success audit (`breakglass.service.ts:390`) runs **before** `signSession` (`:397`), so no token mints if audit throws; `audit_events` has FORCE RLS. **Argon2id** `m=65536/t=3/p=1` (`:31,298`), fresh `randomBytes(16)` salt per rotate (`:474`), `timingSafeEqual` compare (`:303`), lockout checked **before** KDF (`:287`), dummy-hash timing parity on username miss (`:356`).
- **device-add-backdoor.** Silent-add is cryptographically enforced, not policy. (1) `deliverWelcome` requires a **verified existing member** (`messaging.service.ts:238`) and stores only opaque HPKE-sealed Welcome/ratchetTree/commit bytes (`:46-81,228`); the MLS group secret lives only on clients, so a server-inserted `conversation_members` row is pure routing metadata with zero decryption power. (2) The added key is bound to the OOB-verified fingerprint — D1 displays `enrollmentDisplayCode(enrollment.fingerprint)` (`ApproveDevicePanel.tsx:66`) and fans out the key matching that **same** field, re-pinned byte-for-byte (`enroll.ts:59-65`). (3) The signature-key pin **transitively pins** the HPKE keys: ts-mls `createCommit→applyProposals` calls `validateKeyPackage` for every add (`clientState.js:679`), verifying the KeyPackage signature over a TBS containing `initKey` and the leaf signature over a TBS containing `hpkePublicKey`, both under the leaf's own `signaturePublicKey` — so a server that keeps the signature key honest can't swap the encryption key. (4) Approval is real proof-of-possession: `approveEnrollment` verifies an Ed25519 enroll-proof against D1's DB-stored public key (`devices.service.ts:204-210`), requires D1 non-provisional (`:196`), forbids self-approval (`:182`), binds the fresh per-request `enrollmentId`. The only residuals (a fully compromised D1; TOFU if the human skips the compare) are inherent and documented (`multi-device-enrollment.md` T1/T2, §5).
- **fingerprint-binding.** Derivation is over the embedded signature key + identity, never a server routing field: `deviceFingerprint` hashes `leafNode.signaturePublicKey`+`credential.identity` (`index.ts:177-183`); `safetyNumberFromMember` reads `member.signaturePublicKey`+`member.identity` from the validated ratchet-tree roster (`:221-229,542-558`). The 1:1 path has no display-vs-add gap — `prepare()` computes the number over the claimed `peerKeyPackage` (`conversations.ts:156`) and `confirm()` adds that **exact** package (`:214,225`). MITM split is blocked because ts-mls `validateKeyPackage` verifies the leaf self-signature and the whole TBS against the same `leafNode.signaturePublicKey` immediately before the leaf joins the tree; both sides re-derive over the identical committed key (`safety-number.spec.ts:89-114` pins cross-consistency). Enrollment desync closed: register forces `device.signaturePublicKey === fingerprint` (`devices.service.ts:55-57`); fan-out re-pins to that fingerprint and to D1's registered SPK; no path rewrites `devices.signature_public_key` to a different value (key-directory upsert is a same-value no-op, `key-directory.service.ts:98`). **Caveat (F1):** the device-*linking* OOB artifact is the 9-digit `enrollmentDisplayCode`, not the full safety number — see findings.
- **argus-id-identity.** Server-assigned CSPRNG (`argus-id.ts:13-20` via `randomInt`; DB safety-net `gen_argus_id()` rejection-sampling, `0030_argus_id.sql:29-64`), persisted in the challenge row **before** the client sees it, flowing DB→options→verify→insert with **no** inbound `argusId` field on any registration body (`webauthn.controller.ts:201-205,289-293,315-322`). Collision-proof — `UNIQUE` index + 23505-catch-and-regenerate with a loud retry cap (`argus-id.ts:24-44`), never a silent overwrite. Immutable — no inbound schema accepts it (`UpdateProfileSchema` = `displayName`/`avatarSeed` only), and the `users_argus_id_immutable` BEFORE-UPDATE trigger rejects any change server-side (`0030:84-95`). Bound to exactly one tenant from verified auth — `tenantId` derived from `user_tenant_index` keyed by the verified `sub` (`auth.service.ts:69-76`); all three writers use a server-derived `sub`, the table is `SELECT`+`INSERT`-only, `onConflictDoNothing` → first binding wins. Unforgeable `sub` — EdDSA-only mint, `sub` set from the credential-verified caller, never echoed from a request; the WS path re-verifies server-side (`realtime.gateway.ts:104-128`). Discovery is exact-match, tenant-scoped, uniform-null, rate-limited, constant-202, audit-sanitized (`user.service.ts:59-79`, `friends.controller.ts:123-148`).

## Fix-PR disposition

**P1/P2:** none surfaced. No forgery, privilege escalation, cross-tenant, secret-in-log, or admin-path-to-content break was provable.

**Behaviour-touching → each spins off as its own fix PR** (not bundled into the evidence PR):
- **FP-1** — *highest priority.* Widen the enrollment OOB artifact to the full safety-number width; add `enrollmentDisplayCode` coverage. Closes the brute-forceable-backstop MITM.
- **ST-1** — `sid`-revocation cache in `JwtAuthGuard` (or shortened access TTL), *or* an explicit accept-and-document of the 10-min residual in `session-tokens.md`.
- **BG-2** — fail closed in `NODE_ENV=production` when `CF_ACCESS_*` is unarmed.
- **DA-1** — bind a hash of the approved SPK into the enroll-proof and verify it in `approveEnrollment`.
- **DA-2** — KeyPackage-tamper regression test in `packages/crypto`.
- **PK-1** — opt-in DB-integration spec for `webauthn.service.ts`.
- **PK-2** — rename `challenge_hash` → `challenge_hex` (migration; low priority cosmetic).
- **FP-2 / AI-1 / AI-2 / AI-3** — immutability triggers + their DB-integration assertions (group naturally into one or two "identity-column immutability" PRs).

**P3 fix-in-this-PR (doc/comment only — no behaviour change):**
- **ST-2** — claim wording corrected to "user+tenant-bound" (in this note).
- **BG-1** — corrected the runbook UUID in `breakglass-admin.md` **and** `registration-and-tenancy.md`; added a `SELECT`-first confirmation step.

## Guards added / not added (this PR)
- **No new automated guard added in the evidence PR.** The behaviour-touching guards this slice calls for (the four crypto/DB regression tests in DA-2, PK-1, AI-1, AI-2, and the FP-2/AI-3 immutability triggers) are deliberately deferred to their own fix PRs so each ships with the behaviour change it protects.
- **Recommended standing guards** (deferred, high value): (1) the KeyPackage-tamper regression test (DA-2) — it locks the cross-version invariant the *entire* silent-add property rests on; (2) immutability triggers on `external_identity_id` and `signature_public_key` mirroring the existing `argus_id` trigger, each with a `skipIf(!DB_URL)` assertion; (3) a `webauthn.service` DB-integration spec on the `breakglass.service.spec.ts` model.

## Residual risk (accepted for this phase, pending the spun-off PRs)
- **FP-1 device-linking code is the one real exposure.** Until the OOB artifact is widened, a *malicious server* can grind a ~30-bit key collision and inject a device through the linking flow — both the human code-compare and the fan-out byte-pin key off the same server-controlled fingerprint, so they are not independent factors. This is **not** the conversation-add backdoor (that path is separately, cryptographically closed — see `device-add-backdoor`), but it *is* a MITM device-injection surface gated only by a brute-forceable code. **Not acceptable to leave for long — prioritize the FP-1 fix PR.**
- **ST-1 access-token revocation lag** — up to 10 min on non-admin routes after logout/rotate/reuse-detection. Bounded, self-closing, admin surface unaffected; accept-and-document is a legitimate option, but the residual must be written into `session-tokens.md` (it currently documents only the refresh-chain residual).
- **BG-2 unarmed-Access window** — needs *simultaneous* edge + `CF_ACCESS_*` misconfig to bite; compensated by lockout + Argon2id + alertable `breakglass.login_succeeded`. Close it with the prod fail-closed branch.
- **DA-1 / DA-2 cross-version trust** — no current hole; the silent-add property leans on an un-pinned ts-mls invariant. Lock it with the tamper regression test before the next ts-mls bump.
- **Identity-column immutability (FP-2 / AI-1 / AI-2 / AI-3)** — all correctly implemented today, all convention-or-trigger-without-test; future-regression hazards only.
- **Inherent/accepted:** a fully compromised D1 approving a rogue D2, and TOFU first-contact if the human skips the code compare (documented `multi-device-enrollment.md` T1/T2, §5).

## BOTTOM LINE
Auth, identity, and device-trust **hold** — no forgery, privesc, cross-tenant, or admin-path-to-content break was provable, and the **malicious-server-silent-device-add backdoor is closed cryptographically** (the conversation add is bound to the OOB fingerprint + a real proof-of-possession + ts-mls signature validation); the one material gap is **FP-1**, where the separate device-*linking* OOB code is only ~30-bit and server-controlled on both sides, so it must be widened to the full safety number in its own fix PR before the linking flow can be called MITM-safe.
